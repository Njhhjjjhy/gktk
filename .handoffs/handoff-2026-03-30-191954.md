### Phase 5: Mobile tilt module

User provided a complete drop-in mobile tilt module (~300 lines) with three strategies in priority order:

1. **Gyroscope (DeviceOrientation)** — iOS primary. Calls `DeviceOrientationEvent.requestPermission()` on first `touchstart` (iOS 13+ requires user gesture). Maps device beta/gamma to rotationX/Y with baseline captured per touch. MAX_TILT = 25° (reduced from desktop's 40°). Android auto-grants, older iOS tested with 1s timeout.

2. **Touch-drag tilt** — Fallback when gyro is denied/unavailable. Finger position relative to card center drives rotation, same math as desktop mouse but with reduced range. Runs during `touchmove`.

3. **Idle float** — Ambient motion when not touching. Repeating yoyo timeline with randomized ±3-4° rotations on `sine.inOut`, duration 2.5-4s. Stops on touch, resumes after release.

## Problems encountered and how they were solved

### 1. TypeScript error in mobile tilt module
**Symptom:** `pnpm build` failed with "Type 'HTMLElement | null' is not assignable to parameter of type 'HTMLElement'" at line 281 in Orchestrator.tsx.
**Root cause:** `const card = container.querySelector<HTMLElement>(".tilt-card")` returns `HTMLElement | null`. The `handleTouchEnd` closure captured `card` which TypeScript correctly identified as possibly null when passed to `spriteTimelines.get(card)` and `startFloat(card)`.
**Fix:** Added explicit null check `if (activeCard !== card || !card) return;`, used non-null assertion `spriteTimelines.get(card!)`, and captured card reference before delayed callback: `const cardRef = card; gsap.delayedCall(... () => { startFloat(cardRef) })`.

### 2. Card 8 assigned to two types
**Symptom:** User initially assigned card 8 (Exit) to both Type B and Type C.
**Resolution:** Asked for clarification. User corrected: Type B gets cards 2, 8; Type C gets cards 3, 6, 7.

### 3. Content-specific CSS removed during comparison rewrite
**Symptom:** When replacing the 8-card presentation section with the 3-card comparison layout, all content-specific styles were removed (product grid, IRR data rows, risk items).
**Resolution:** Had to re-add all content-specific CSS when rebuilding the 8-card production layout. Styles were adapted for the new dark-themed Type A cards (IRR data rows needed light-on-dark color scheme instead of the original light-card colors).

### 4. Fixed height vs min-height for content cards

## Learnings and discoveries

1. **Type A (background + overlay) forces dark theme.** When cycling image layers are behind content, text must be light regardless of the card's intended color scheme. The original IRR cards were light-themed but became dark-themed in Type A. This required dark variants for all data row styles (labels → neutral-300, lines → rgba white, highlight box → semi-transparent white bg).

2. **Type C is the most flexible.** It accepts children directly and lets them control their own layout with `<CyclingLayers />` placed wherever needed. This worked for both product grids (cycling images in each product slot) and risk items (cycling image banners above each risk). Type A and B are more opinionated about structure.

3. **The reference site's observe-based card swap avoids the tilt/scrub conflict.** The previous scrub-based card stack used rotationX for flip-in/flip-out, which conflicted with the tilt's rotationX. The reference uses `ScrollTrigger.observe` with fling-to-dismiss (yPercent: -300) which doesn't touch rotationX at all. The current vertical scroll layout sidesteps this conflict entirely since there's no card-stack animation.

4. **Mobile tilt gyroscope permission must be in a user gesture.** iOS 13+ requires `DeviceOrientationEvent.requestPermission()` to be called inside a touch/click handler. The module correctly places it in `touchstart`. It only requests once — if denied, falls through to touch-drag permanently.

5. **The `{ passive: true }` flag is critical on iOS Safari.** Without it, touch listeners trigger Safari's intervention warning about non-passive event listeners blocking scroll. All touch handlers in the mobile module use passive mode.

6. **Risk cards benefit from wide cycling images.** The `.tilt-card__risk-image` uses 21:9 aspect ratio for a cinematic banner feel above each risk item. On mobile this shifts to 16:9 for better vertical space usage.

7. **The user wants approaches kept as named, reusable components.** They explicitly said "keep them as components" referring to A, B, C. This led to the TiltCards.tsx component file with exported `TiltCardA`, `TiltCardB`, `TiltCardC`, and `CyclingLayers`.

## Architecture and design notes

### Three tilt-card types as components

The site uses three reusable card components, each combining content with cycling image layers differently:

**Type A (`TiltCardA`):** Full-bleed cycling image layers as card background. Dark gradient overlay (92% opacity at bottom → 25% at top) ensures text readability. Content sits on top at z-index 2. Always dark-themed. Best for: text-focused cards where imagery is atmospheric (concept overview, financial data). Props: `align` controls whether content pins to bottom (default, for short content) or top (for data-heavy content).

**Type B (`TiltCardB`):** Split layout — text on left, cycling image stack filling the right column. Clear separation between content and imagery. Supports light or dark variants. Best for: narrative cards with a single visual focus (persona, exit strategy).

**Type C (`TiltCardC`):** Flexible content wrapper. Cycling images placed by children using `<CyclingLayers />` in individual image slots (grid items, product cards, risk banners). Supports light or dark variants. Best for: structured content with multiple image areas (product grid, risk items).

**`CyclingLayers`:** Shared component rendering 5 absolutely-positioned `.tilt-card__layer` elements inside a `[data-cycling]` container. The Orchestrator creates GSAP timelines for each `[data-cycling]` element: `autoAlpha: 1, repeat: -1, duration: 0, repeatDelay: 1, stagger: {amount: 3}, yoyo: true`. Timelines are paused by default, played on hover (desktop) or touch (mobile).

### Vertical scroll vs pinned card stack

The current layout uses simple vertical scroll — each card in its own 100vh `.presentation__card` container. This was chosen over the pinned card stack because:
1. The 3D tilt interaction works without conflict (no scrub-driven rotationX)
2. Anchor navigation works correctly (`lenis.scrollTo(element)` targets real DOM positions)

4. Simpler to reason about and debug

### Mobile tilt architecture

Three strategies in priority order, all inside `initMobileTilt()`:

1. **Gyroscope:** `DeviceOrientationEvent` with iOS permission flow. Beta → rotationX, gamma → rotationY. Baseline captured per touch. MAX_TILT 25°. Duration 0.5s, power2.out easing.

2. **Touch-drag:** Finger position relative to card center mapped to rotation. Same formula as desktop mouse. Fallback when gyro unavailable/denied.

3. **Idle float:** Repeating yoyo timeline with random small rotations (±3-4°) on sine.inOut. Duration 2.5-4s. Stops on touch, resumes after release settle (elastic.out spring).

Touch behaviors: scale 1.08 on press (back.out(1.7)), 0.95 press effect, elastic.out(1, 0.5) release spring. Layer cycling: faster than desktop (0.6s repeat, 2s stagger). All listeners passive.

### Image layer system

Each `[data-cycling]` container holds 5 `.tilt-card__layer` children. Layers are absolutely positioned, filling their parent. First layer starts at opacity 1, others at 0. GSAP cycles through them with instant autoAlpha swaps and yoyo repeat. When real images replace the gradient placeholders, each layer div becomes an `<img>` tag with `object-fit: cover`.

The same layer system works in all three card types:
- Type A: layers fill the entire card (background)
- Type B: layers fill the `.tilt-card__split-image` area
- Type C: layers fill individual `.tilt-card__product-image`, `.tilt-card__risk-image`, or `.tilt-card__grid-image` slots
