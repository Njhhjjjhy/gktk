## Problems encountered and how they were solved

### 1. vh-based card heights don't work across devices
**Symptom:** Cards too large on iPad (tall viewport), too small on desktop (wide viewport).
**Root cause:** `vh` is relative to viewport height, which varies wildly across devices.
**Fix:** Used fixed pixel heights: 420px mobile, 480px tablet, 540px desktop.

### 2. CardA dark background made text invisible
**Symptom:** All text on IRR cards was nearly invisible — dark text on dark background.
**Root cause:** CardA has CyclingLayers (dark gradient backgrounds) + `.card__gradient` overlay. After changing all text colors to dark neutrals (neutral-700, 950, 900), there was no contrast.
**Fix:** Tried adding `.card__inner--overlay` color overrides for light text, but user rejected this because it was inconsistent with other cards. Switched CardA usages to CardB (white background). CardA is now unused.

### 3. Hero centering creates dead space
**Symptom:** Large empty area between nav and hero content, especially on mobile.
**Root cause:** `min-height: 100vh` with flex centering pushes short content to the vertical middle, leaving ~40% empty space above. The fixed nav adds complexity because `padding-top` for nav clearance shifts the center point.
**Fix:** UNRESOLVED. Multiple approaches tried (see Phase 8 above). Current code uses `min-height: 100vh` + flex centering on all breakpoints. The user is not satisfied.

### 4. Hero spacing inconsistency
**Symptom:** Gaps between hero heading/subheading/body were much larger than card gaps.
**Root cause:** Hero originally used a single `gap` on `.hero__content` flex container (1.953rem = 31px). Card gaps were 8px/12px.
**Fix:** Removed `gap`, added individual `margin-bottom` values: heading 8/10/12px, subheading 16/18/20px.

### 5. Showcase docs reverted after stash/pop
**Symptom:** After `git stash pop`, some showcase docs showed old tilt content.
**Root cause:** The showcase doc updates were done by a background agent. When the user restarted VS Code and stash-popped, some files may have been modified by a linter or the IDE.
**Fix:** The system reminders show these files were "modified by user or linter" — the current state needs verification.

## Architecture and design notes

### Card system architecture
All cards use the same base structure: `.cards-section__card > .card-container > .card.card--dark > .card__inner > [content]`. The `.card` has a fixed height per breakpoint, `.card__inner` fills it with `height: 100%` and `overflow-y: auto` (hidden scrollbar). Content flows from top.

Three component types exist (CardA, CardB, CardC) but only B and C are used. B and C are identical in markup — both render `div.card.card--dark > div.card__inner > {children}`. The distinction is conceptual only.

### Spacing system (mobile-first, scaling up)
The user established a strict spacing system that should apply everywhere:
- label → title: 8px / 10px / 12px
- title → body: 12px / 14px / 16px (cards) or subheading → body: 16px / 18px / 20px (hero)
- body → image: 16px / 20px / 24px
- Card inner padding: 24px / 28px / 32px
- Card outer padding: 48px 24px / 56px 32px / 64px 40px

### Typography system (mobile-first)
- Section label: Noto Sans JP 400, 14px / 16px / 20px, neutral-700
- Card title: REM 600, 20px / 25px / 31px (section-heading tokens), neutral-950
- Card body: Noto Sans JP 400, 16px / 20px / 25px (section-body tokens), neutral-900
- Data group-label: REM 400, 14px / 16px / 20px, neutral-600
- Data row-label: Noto Sans JP 400, 14px / 16px / 20px, neutral-600
- Data row-value: Noto Sans JP 600, 14px / 16px / 20px, neutral-800

### Design tokens
Colors, radii, type scale values are in `:root` in globals.css lines 9-82. Breakpoints: mobile ≤744px, tablet 745-1024px, desktop ≥1025px. Major-third (1.25) type scale.

### Nav is position: fixed
The nav overlaps page content. All sections must account for nav height via padding-top. `--nav-height` ≈ 77px desktop, `--nav-height-mobile` = 48px.
