## Problems encountered and how they were solved

### 1. No problems with compilation
All three complete file rewrites compiled on first attempt with `next build`. TypeScript passed, no runtime errors detected at build time.

### 2. Concept chapter cycling removed without replacement
**Symptom:** The concept section previously had 3 cycling chapters with auto-play interval and stepper buttons. In the tilt-card restructure, all 3 chapters are shown simultaneously as a grid.
**Resolution:** This was intentional — the tilt-card format shows all content at once rather than cycling through it. The stepper UI and interval logic were removed from Orchestrator. No cycling needed when content is visible in a grid.

### 3. Risk accordion removed without replacement
**Symptom:** The risk section previously had expandable accordion items. In the tilt-card restructure, risk content is shown flat.
**Resolution:** Intentional. Accordion interaction doesn't work within a scrub-driven card stack (user can't click to expand while scrolling through cards). Content is now fully visible with risk text and hedge side-by-side in a 2-column grid per item.

### 4. Anchor navigation may not work correctly with pinned section
**Symptom:** Nav links use `href="#concept"`, `#persona"`, etc. These IDs are now on tilt-card elements WITHIN the pinned presentation section. Scrolling to `#concept` would scroll to the top of the presentation section, not to the specific card.
**Root cause:** With a pinned card stack, all cards occupy the same viewport position. Browser hash navigation and `element.scrollIntoView()` would scroll to the presentation section top, not to the card's position within the scrub timeline.
**Resolution:** NOT YET FIXED. The `handleAnchorClick` in Orchestrator uses `lenis.scrollTo(target)` which would scroll to the element's DOM position. This works for elements in normal flow (hero, footer) but for cards within the pinned section, it would scroll to the pinned section top regardless of which card ID was targeted. A proper fix would calculate the scroll position based on each card's index within the timeline and use `scrollTo` with a specific pixel value.
**Future note:** To fix, calculate each card's scroll position: `presentationST.start + (cardIndex / totalCards) * (presentationST.end - presentationST.start)` and use `lenis.scrollTo(calculatedPosition)`.

## Learnings and discoveries

1. **The user treats the reference spec as authoritative law.** When the user says "do what the prompt I just gave you," they mean apply EVERY structural pattern described, not selectively choose which to implement. Match the reference's architecture, not just individual properties.

2. **The 0.8333vw body font-size is NEVER overridden.** This is the foundation of the reference site's scaling system. Mobile overrides use rem, px, or large em multipliers to compensate for the tiny em values (3-4px per em at mobile viewports). Previous sessions incorrectly switched to `font-size: 1rem` on mobile.

3. **clip-path: inset(0 0 0 0) is preferred over overflow: hidden** for text mask containers because it doesn't interfere with 3D transforms (rotationX, perspective, etc.). The card-stack uses rotationX extensively, so this is important.

4. **Lenis only needs touchMultiplier: 0.2.** All other parameters (lerp: 0.1, duration: 1, smoothWheel: true, etc.) are Lenis defaults. Setting them explicitly is redundant.

5. **The preloader digit columns scroll at DIFFERENT speeds.** Ones column is fastest (power4.inOut), tens is medium (power3.inOut), hundreds is slowest (power2.inOut with delay). This creates the mechanical counter feel of the reference.

6. **The reference hero entrance plays ALL elements simultaneously at position 0.** Hero mask scaleY→0, title reveals, nav entrance, and background all start at the same time — NOT sequentially. This creates a dramatic "everything reveals at once" moment.

7. **Risk content on tilt-cards must be flat, not accordion.** Accordion interaction is incompatible with scrub-driven card stacks — the user can't click to expand while the page is scroll-locked to the card-stack scrub.

8. **The tilt-card name is important to the user.** They explicitly said "KEEP THIS NAME" when describing the cards. Use `tilt-card` in class names and discussions, not "slide" or "card."

9. **Content-heavy sections should split into multiple tilt-cards.** The user explicitly instructed: "if there is too much content in 1 section that make it another tilt-card ex: tilt-card-concept-1, tilt-card-concept-2." IRR and Risk were each split into 2 cards.

10. **Mobile card-stack should break into vertical flow.** A 1000vh pinned section with scrub doesn't work on mobile touch. The mobile breakpoint converts the card stack into a normal flex column with each card at min-height: 100svh.

## Architecture and design notes

### Tilt-card presentation architecture
The entire site content (except hero and footer) lives within a single pinned card-stack section. This replaces the previous architecture of separate full-viewport sections. The structure is:

```
Hero (100vh, 3-row grid)
└── Presentation section
    └── .presentation__wrapper (1000vh scroll-driver)
        └── .presentation__slides (100vh, pinned, grid 1/1)
            ├── .tilt-card--first (concept, light bg, bg scales from 0.075)
            ├── .tilt-card--stack (persona, dark)
            ├── .tilt-card--stack (product, dark)
            ├── .tilt-card--stack (irr-1, light)
            ├── .tilt-card--stack (irr-2, light)
            ├── .tilt-card--stack (risk-1, dark)
            ├── .tilt-card--stack (risk-2, dark)
            └── .tilt-card--stack (exit, light)
Footer (100vh)
```

All cards stack in the same grid cell (grid-area: 1/1). GSAP ScrollTrigger pins the slides container and scrubs through the card-stack timeline as the user scrolls through the 1000vh wrapper.

### Card-stack animation timeline
- First card: bg scales from 0.075 → 1 (0→0.5), holds (0.5→1.0), exits with rotationX: -40, y: -6%, scale: 0.7 (1.0→1.5)
- Each stack card: enters with rotationX: 40→0, scale: 1.1→1 (duration 1.0), exits with rotationX: 0→-40 (duration 0.5, except last card which has no exit)
- Enter start position: 1.1 + i * 1.5 (where i is the stack card index 0-6)
- Total approximate timeline duration: ~12.1 normalized units
- Effective scroll per card: ~83vh

### Tilt-card CSS class convention
- `.tilt-card` — base (width 89.79%, border-radius 1.5rem, clip-path)
- `.tilt-card--first` — transparent bg, uses `.tilt-card__bg` child for scale animation
- `.tilt-card--stack` — standard stack card (GSAP controls entrance/exit)
- `.tilt-card--light` / `.tilt-card--dark` — color variants
- `.tilt-card__inner` — content container (padding 4em, flex column)
- `.tilt-card__inner--top` — top-aligned variant for data-heavy cards (IRR, risk)
- `.tilt-card__split` — 2-column layout (persona, exit)
- `.tilt-card__grid` — 3-column layout (concept)
- `.tilt-card__product-grid` — 3×2 image grid (product)
- `.tilt-card__group` / `__row` / `__row-label` / `__row-line` / `__row-value` — IRR data rows
- `.tilt-card__risk-item` / `__risk-label` / `__risk-body` / `__hedge-label` — risk items (flat layout)

### Dark/light rhythm
Concept (light) → Persona (dark) → Product (dark) → IRR-1 (light) → IRR-2 (light) → Risk-1 (dark) → Risk-2 (dark) → Exit (light). The dark presentation wrapper background shows through the rounded corners of light cards, creating a frame effect.

### Mobile strategy
On mobile (≤767px), the pinned card-stack breaks into a vertical flex column. Each card becomes full-width with min-height: 100svh. No pinning, no scrub, no rotationX animations. Cards simply stack vertically with 0.75rem gap. All grids collapse to 1 or 2 columns. Typography switches to rem/px fixed sizes.

### Scaling system
Body font-size: 0.8333vw at ALL viewports. At 1440px → 1em = 12px. At 375px → 1em = 3.125px. Desktop styles use em (scales proportionally). Mobile overrides use rem/px (fixed sizes independent of the tiny em). This is the reference site's exact approach — the vw base is never overridden.
