## Architecture and design notes

### The em scaling system -- understanding the problem

The site uses `body { font-size: 0.8333333333vw }` (line 60 of globals.css), which makes `1em` equal to approximately:
- 1920px viewport: ~16px
- 1440px viewport: ~12px
- 768px viewport: ~6.4px
- 375px viewport: ~3.1px

All `em`-based dimensions in the tilt-card system (container width `72em`, inner padding `4em`, title `5em`, grid gaps `3em`, etc.) scale proportionally with viewport width. This creates a responsive scaling effect but makes the cards very different sizes at different viewports.

The reference site (radiance.family) used a similar trick but with an even tinier body font-size (~4.19px on mobile), and its cards contained only images -- no dense text content. GKTK's cards contain data tables, multi-column grids, and paragraphs, so they need more space than the reference's 384x464px.

**Key tension:** The em-based scaling is elegant for responsive design, but the card dimensions don't match the reference site's intended sizes. A pure px conversion makes content illegibly small. The right approach likely involves:
- Keeping viewport-relative scaling for typography and spacing (em/rem)
- Using fixed dimensions (px or CSS custom properties) only for the card container width/height
- Possibly using `clamp()` for elements that should scale but within bounds

This architectural question remains unresolved.
